# 为什么PM需要学习Vibe Coding

## 能力边界的扩展

### 传统PM工作流

```mermaid
flowchart LR
    A[想法] --> B[PRD]
    B --> C[交付研发]
    C --> D[等待开发]
    D --> E[验收测试]
    E --> F[上线]

    C -.-> |依赖研发资源| D
    D -.-> |周期：数周| E
```

### Vibe Coding PM工作流

```mermaid
flowchart LR
    A[想法] --> B[自己做原型]
    B --> C[快速验证]
    C --> D[精准PRD]
    D --> E[研发实现]

    B -.-> |自主可控| C
    C -.-> |周期：数小时| D
```

## 核心价值

> [!tip] McKinsey 调研（2024）
> 在软件相关的“内容密集型”任务上，生成式 AI 可节省约 **40%** 时间；“内容较轻”任务可节省约 **15%** 时间，综合流程约 **30%** 时间下降（185→129 分钟）。

### 场景对比

| 场景 | 传统方式 | Vibe Coding 方式 |
|:---|:---|:---|
| **验证想法** | 等研发排期做demo | ==自己30分钟出原型== |
| **沟通需求** | 文字描述+原型图 | 可运行的demo+代码 |
| **用户测试** | 等待开发完成 | 快速迭代多版本 |
| **技术评估** | 依赖研发判断 | 自己尝试后有体感 |

### 价值矩阵

```
高价值 ┃ ★ 快速原型      ★ 用户验证
       ┃   (小时级)        (自主可控)
───────╋───────────────────────────────
       ┃ ○ 技术沟通      ○ 需求精准
低价值 ┃   (理解更深)      (细节到位)
       ┃
       └───────────────────────────────
           低频使用          高频使用
```

## 具体应用场景

### 适合PM自己做的事

> [!success] 推荐场景
>
> **1. 快速原型验证**
> - 用 Lovable 快速做出可交互原型
> - 直接给用户测试，收集反馈
>
> **2. 内部工具**
> - Dashboard 数据看板
> - Admin 后台管理
> - 自动化脚本
>
> **3. Demo 制作**
> - 给领导汇报的演示
> - 给客户展示的原型
> - 用研时的测试工具

### 仍需工程师参与的事

> [!warning] 需要支持
>
> - 涉及用户数据安全
> - 支付相关功能

## 学习投入产出分析

### 时间投入

| 阶段 | 时间 | 产出 |
|:---|:---:|:---|
| 入门 | 1周 | 能用在线平台做简单原型 |
| 进阶 | 2-3周 | 能用IDE工具做完整Demo |
| 精通 | 1月+ | 能独立完成复杂原型 |

### ROI 估算

```
假设：
- PM 平均等研发排期时间：5天
- 使用 Vibe Coding 自己做：0.5天
- 每月平均 4 个验证需求

节省时间 = (5 - 0.5) × 4 = 18天/月

学习投入 = 2周（一次性）
回报周期 < 1个月
```

## 常见顾虑解答

> [!question] Q: 我没有编程基础，能学会吗？
> **A:** 可以！Vibe Coding 的核心是==自然语言描述需求==，不需要掌握编程语法。从 Lovable、v0 等在线平台开始，零代码基础也能上手。

> [!question] Q: 这会取代工程师吗？
> **A:** 不会。Vibe Coding 适合原型和验证，生产级系统仍需专业工程师。这是==赋能PM==，不是替代工程师。

> [!question] Q: 生成的代码质量如何？
> **A:** 对于原型来说足够用。关键是理解：原型可以"够用就好"，生产代码需要工程师把关。
